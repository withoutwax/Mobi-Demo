# Comprehensive Test Specification

이 문서는 모빌리티 데이터 파이프라인(Mobi-Demo)의 각 기능 단위별 테스트 시나리오를 명세합니다.
프로젝트 규칙에 따라 각 핵심 기능별로 최소 20개의 단위/통합 테스트 케이스를 도출하여 극단적인 예외 상황, 엣지 케이스, 경계값, 병렬성 등을 촘촘하게 검증합니다.

## 1. Domain Model (`VehicleDomainTest`) - 목표: 20개

### 다형성 및 타입 검증 (5)

- [ ] 1. EvVehicle 인스턴스 생성 시 타입이 EV 파워트레인으로 지정되는지 검증
- [ ] 2. IceVehicle 인스턴스 생성 시 타입이 ICE 파워트레인으로 지정되는지 검증
- [ ] 3. VehicleType.REGULAR로 지정 시 정상 생성 검증
- [ ] 4. VehicleType.FREIGHT로 지정 시 정상 생성 검증
- [ ] 5. VehicleType.MICRO로 지정 (EvVehicle) 시 정상 생성 검증

### 제약사항 및 예외 방어 (Fail-fast) 검증 (6)

- [ ] 6. 불변식 위반: IceVehicle에 MICRO 타입을 지정 시 IllegalArgumentException 발생
- [ ] 7. EvVehicle의 배터리 레벨이 0.0일 때 무사 통과 (경계값)
- [ ] 8. EvVehicle의 배터리 레벨이 100.0일 때 무사 통과 (경계값)
- [ ] 9. EvVehicle의 배터리 레벨이 -0.1일 때 예외 발생 (음수 불가)
- [ ] 10. EvVehicle의 배터리 레벨이 100.1일 때 예외 발생 (100 초과 불가)
- [ ] 11. IceVehicle의 연료 레벨이 허용 구간을 벗어났을 때 예외 발생 (-1.0, 101.0 등)

### 필드 정합성 및 Kotlin Data Class 특성 검증 (6)

- [ ] 12. 공통 프로퍼티(latitude, longitude, speed, temperature, vehicleId) 접근 검증
- [ ] 13. EvVehicle 객체의 동등성(equals) 검증 (데이터가 같으면 true)
- [ ] 14. IceVehicle 객체의 동등성(equals) 검증
- [ ] 15. EvVehicle 객체의 copy()를 통한 불변 객체 복사 검증
- [ ] 16. copy() 시나리오: ID만 변경했을 때 나머지가 유지되고 동등성은 false인지 검증
- [ ] 17. hashCode() 구현 규칙 검증: 동일한 데이터를 가진 두 인스턴스가 같은 해시코드를 반환하는지 검증

### 특수한 상태 (Edge Case) (3)

- [ ] 18. 위도/경도가 극단적인 값(음수 등)이 입력되어도 도메인 객체 자체는 생성되는지 확인 (현재 도메인에 좌표 제약 없음)
- [ ] 19. 속도(speed)가 0인 정차 상태 생성 검증
- [ ] 20. 차량 ID가 빈 문자열("")일 때 생성 동작 확인 (필요시 도메인 로직에 제약 추가 제안용)

- [ ] 20. 반환된 타입(리스트)에 null 요소가 포함되지 않고 완전한 데이터를 갖는지 컬렉션 정합성 검증

## 3. Buffer (`VehicleDataBufferTest` & Concurrency) - 목표: 20개

### 기본 Queue 동작 검증 (5)

- [ ] 1. 초기 생성된 버퍼의 size()가 0인지 검증
- [ ] 2. 버퍼가 비어있을 때 poll() 호출 시 예외 없이 null을 반환하는가
- [ ] 3. 단일 차량 데이터 offer() 시 예외가 발생하지 않고 크기가 1이 되는지 검증
- [ ] 4. offer()로を入れた 순서(FIFO)대로 poll()에서 데이터가 꺼내어지는지 검증
- [ ] 5. offerAll() 호출 시 복수 개의 리스트가 큐 맨 뒤에 일괄 정상 삽입되는지 검증

### 최대 용량 (Capacity) 및 예외 처리 (5)

- [ ] 6. 버퍼 크기가 1000에 도달할 때까지 offer() 성공 검증
- [ ] 7. 용량이 1000인 버퍼에 1001번째 offer() 시도 시 IllegalStateException 예외 확인
- [ ] 8. 가득 찬 버퍼에서 poll()을 1회 호출해 여유 공간을 1 늘린 후 다시 offer()하면 성공하는지 검증
- [ ] 9. offerAll() 수행 시 (현재 사이즈 + 넣을 리스트 사이즈 > 1000)일 때 앞부분만 들어가다 예외가 발생하는지 과정 검증
- [ ] 10. 1000개 꽉 찬 버퍼를 while(poll() != null)로 완전히 비울 경우 size가 0이 되는지 확인

### 동시성 및 다중 스레드 안전성 (Race Condition) (5)

- [ ] 11. 10개 스레드가 각각 100번씩 offer() 호출 시 최종 사이즈가 명확히 1000인지 확인 (유실 부재)
- [ ] 12. 1000개의 데이터가 있는 상태에서 100개 스레드가 각각 10회씩 poll() 호출 시 잔여 0인지 확인
- [ ] 13. [기존 포함] 생산자(50) + 소비자(50) 동시성 교차 작업 시 데드락 없음 검증
- [ ] 14. 멀티스레드 offer 중 1000개 초과 시 일부 스레드만 정상적으로 예외(IllegalStateException)를 맞는지 검증
- [ ] 15. 꺼낼 데이터가 부족할 때 멀티스레드 poll()이 블로킹되지 않고 안전하게 null을 반환하는지 검증

### 엣지 및 성능 확인용 (5)

- [ ] 16. 극단적으로 많은 offerAll()과 빈번한 poll() 혼합 워크로드에서의 안전성 테스트
- [ ] 17. offer()에 올바른 타입의 Vehicle 인스턴스(Ev, Ice 모두)를 무작위로 섞어 넣었을 때 타입 손실이 없는지 확인
- [ ] 18. 버퍼의 참조 객체가 끊어질 수 있는가? - 메모리 릭(Memory Leak) 관점에서 내부 노드 해제 동작 유추 테스트
- [ ] 19. poll() 이후 객체 참조가 버퍼에 남아있지 않는지 사이즈 변화로 교차 검증
- [ ] 20. 여러 연속된 호출 시 초당 수 만 건의 데이터가 들어와도 ConcurrentModification Exception 발생 여부 (발생하지 않음 증명)

## 4. Producer (`VehicleDataProducerTest`) - 목표: 20개

### 정상 스케줄링 흐름 검증 (5)

- [ ] 1. produceData() 호출 시 generator.generate()가 정확히 1회 호출되는가
- [ ] 2. generator가 생성한 데이터 리스트가 정확히 buffer.offerAll()에 1구간 전달되는가
- [ ] 3. 반환된 차량 리스트의 개수가 5일 때, offerAll에 5개짜리 리스트가 들어가는지 Capture 검증
- [ ] 4. 반환된 리스트가 0개일 때(가정), offerAll이 empty list를 받거나 예외 없이 통과되는지 확인
- [ ] 5. produceData()를 연속 3번 호출 시 각각 독립적으로 generate()와 buffer.offerAll()이 3회씩 호출되는지 검증

### 의존성 장애 시의 스레드 생존성 (Fault Tolerance) (8)

- [ ] 6. buffer.offerAll()에서 `IllegalStateException`(버퍼 초과) 발생 시 produceData()가 죽지 않음 검증
- [ ] 7. 버퍼 초과 발생 후, 두 번째 produceData() 호출 시 정상적으로 다시 generate()와 offerAll()을 시도하는지 검증 (회복 체력)
- [ ] 8. generator.generate() 자체에서 RuntimeException 발생 시 스케줄러가 잡아서 죽지 않는지 검증
- [ ] 9. buffer 측에서 NullPointerException이나 알 수 없는 예외 투척 시 안전망이 동작하는지 확인
- [ ] 10. 외부 로그 라이브러리 연동 시 `log.warn`이나 `log.error` 메서드가 올바르게 인터셉트/호출되는지 텍스트 캡처 테스트
- [ ] 11. 부분 장애: generate()에서 예외 발생 시 buffer 연산은 시도 자체가 안 일어나는지 검증
- [ ] 12. 정상 생성 후 offerAll에서만 예외 날 때도 다음 주기 실행이 보장되는 트랜잭션 단위 파악 테스트
- [ ] 13. 모든 의존성이 최악의 에러(OutOfMemoryError 가상 모방 예외)를 던져도 최상위 스프링 스케줄링 스레드에 영향 없는지 테스트

### 논리적 타이밍 및 Mock 상호작용 (7)

- [ ] 14. produce() 로직 수행 시간이 매우 길게 모의(Mocking delay)되었을 때 타임아웃 예외가 스레드에 던져지는지 확인
- [ ] 15. 한 주기에서 offerAll 시 인자로 넘어간 데이터의 인스턴스가 generator가 내뱉은 객체 레퍼런스와 100% 일치하는지 참조 동등성 검증
- [ ] 16. producer 인스턴스 초기화 시 인자로 null이 투입될 경우의 Kotlin 널안전성 파악 (Mock 간접 파악)
- [ ] 17. 생성기가 항상 똑같은 객체를 반환하는 Mock 환경이어도 버퍼가 알아서 받는 행위가 보장되는지 검증
- [ ] 18. 예외 없이 정상 처리되었을 때 `log.debug`가 정상 호출된 기록이 남는지 로그 Appender 검증
- [ ] 19. generator의 반환 속도가 순간적으로 높을 때(차량 10만 대 반환) producer 로직 자체에는 메모리 부하 외에 코드 결함 여부 확인
- [ ] 20. 매 호출 시마다 새로운 예외 유형(1차 IllegaState, 2차 Runtime)을 번갈아 발생시켜도 회복력이 있는지 확인

## 5. SseEmitterService (`SseEmitterServiceTest`) - 목표: 20개

### 구독(Subscribe) 및 해제 관리 (6)

- [ ] 1. subscribe() 호출 시 60,000ms 타임아웃의 Emitter가 반환되는지 확인 (Reflection 등 활용)
- [ ] 2. subscribe() 직후 내부 커넥션 카운트(getEmitterCount)가 정확히 1 증가하는지 확인
- [ ] 3. subscribe() 직후 "INIT" 더미 이벤트가 클라이언트로 1회 즉시 전송되는지 커넥션 검증
- [ ] 4. Emitter의 onCompletion 콜백이 트리거되었을 때, 리스트에서 해당 클라이언트가 제거되는지 검증
- [ ] 5. Emitter의 onTimeout 콜백이 트리거되었을 때, `.complete()` 호출과 함께 리스트에서 제거되는지 확인
- [ ] 6. onError 콜백 트리거 시 `.completeWithError()` 호출 후 안전하게 리스트 제거 검증

### 브로드캐스팅(Broadcast) 및 정상 전송 (5)

- [ ] 7. 브로드캐스트 호출 시 0명일 때 아무 동작도 하지 않는지 검증 (빠른 return)
- [ ] 8. 1명의 정상 클라이언트에게 10개의 Vehicle 데이터를 보냈을 때 send() 메서드가 정확히 데이터를 캡슐화한 이벤트를 보내는지 확인
- [ ] 9. 10명의 클라이언트가 존재할 때, 각각의 Emitter.send()가 정확히 1회씩 도합 10회 호출되는지 검증
- [ ] 10. 전송되는 이벤트의 Name 속성이 정확히 "vehicles" 라는 문자열로 설정되었는지 ArgumentCapture 로 검증
- [ ] 11. 브로드캐스트 리스트에 빈 리스트를 전달하더라도 에러 없이 "빈 배열"이 그대로 각 Emitter에 방송되는 지 판단

### 자가 치유(Self-Healing) 방어 로직 검증 (6)

- [ ] 12. 전송 중 특정 Emitter에서 IOException 발생 시, 해당 Emitter만 로그를 남기고 리스트에서 즉각 삭제되는지 확인
- [ ] 13. [기존 통합] 1정상 + 1에러 클라이언트 중, 1정상은 전송되고 1에러는 삭제됨을 확인
- [ ] 14. 10명 중 앞 5명이 차례로 예외를 던져도 뒤 5명은 send()를 무사히 전달 받는지 (반복문 중단 불가) 확인
- [ ] 15. Emitter에서 IllegalStateException (이미 complete된 상태에서 send) 시에도 서버가 죽지 않고 remove 됨을 검증
- [ ] 16. 초기 subscribe() 중에 "INIT" 이벤트 전송 시 IOException이 나면 해당 Emitter를 즉시 없애고, 버려진 상태로 반환되는지 확인
- [ ] 17. NullPointerException 과 같은 예기치 못한 Exception을 `send()`에서 뱉더라도 꿀꺽 잡아서 치유하는지 검증

### 동시성 및 병렬 스레드 조작 (3)

- [ ] 18. 100명의 유저가 멀티스레드에서 동시에 subscribe()를 요청할 때, 컬렉션이 사이즈 100으로 딱 물리게 동작하는지 (CopyOnWriteArrayList 안전성) 확인
- [ ] 19. 브로드캐스팅(순회) 작업이 돌아가는 바로 그 찰나에 신규 클라이언트가 subscribe()로 추가되거나 onError로 나갔을 때 CME가 발생하지 않는지 검증
- [ ] 20. 순회 도중에 클라이언트 삭제(removeEmitterForTest 조작)가 일어나도 정상 클라이언트에 대한 남은 순회를 완주하는지 증명

## 6. Broadcaster (`VehicleDataBroadcasterTest`) - 목표: 20개

### 조기 반환 (Early Return / Connection-Aware) 검증 (5)

- [ ] 1. 접속자가 0명일 때 broadcastData() 호출 시 buffer.poll()이 단 1회도 이루어지지 않는지 검증
- [ ] 2. 접속자가 0명일 때 sseService.broadcast()가 절대로 호출되지 않는지(성능 낭비 없음) 검증
- [ ] 3. [에지 케이스] 접속자 카운트가 에러 나 음수(-1)로 반환될 리 없으나 0 포함 "1 이상이 아니면" 안 타게 하는 로직인지 파악
- [ ] 4. 접속자가 0명이라 조기 반환 시, 이 과정 중 런타임 에러가 숨어 있어도 try-catch가 먹히는지
- [ ] 5. 접속자 수를 체크하는 `getEmitterCount()` 메서드는 정확히 1회 호출되어 비용을 아끼는지 검증

### 폴링(Polling) 및 큐(Drain) 동작 검증 (5)

- [ ] 6. 접속자 1명 이상일 때, 버퍼에 데이터가 10개 있으면 poll()이 정확히 11회(10회 리턴, 1회 null 탈출) 호출되는가
- [ ] 7. 단일 큐 사이클 내에서 꺼내진 모든 데이터 순서가 바뀌지 않고 배열에 축적되어 브로드캐스트로 넘겨지는가 (동등성 확인)
- [ ] 8. 접속자가 존재하지만 큐가 완전히 비어있어(poll 즉각 null) 꺼낸 데이터가 0개일 때, `sseService.broadcast()`는 **생략**하는 불필요 전송 차단 로직이 구현되었는지 확인
- [ ] 9. 버퍼에 1000개가 있어도 무한루프(while)를 통한 완전 소진 후 List 크기 1000으로 만들어서 넘기는지 볼륨 검증
- [ ] 10. poll() 시 만약 버퍼 측 내부 에러(Null 예외 등)가 나도 브로드캐스터가 죽지 않는지 방어력 검증

### SSE 전송 연동 검증 (6)

- [ ] 11. 최종 추출된 리스트를 인자로 하여 `sseService.broadcast(list)`가 정확히 1번만 일괄 전송 호출되는지 확인
- [ ] 12. 다수의 클라이언트(예: 100명)일 때도 로직 자체는 `SseEmitterService` 내부에 위임하므로 broadcaster는 여전히 역할을 동일하게(1회 호출) 완수하는가
- [ ] 13. [기존] broadcast 도중 sseService 측에서 런타임 익셉션을 던져도 워커(Scheduled) 스레드가 뻗지 않음을 검증 (catch)
- [ ] 14. sseService.broadcast(list) 가 호출되는 데이터 리스트 인스턴스가 10개 요소로 매핑이 정확한지 ArgumentCaptor 검증
- [ ] 15. 이 빈이 스프링 내 애플리케이션 컨텍스트에서 @Scheduled 와 조합 시 실제 어떤 딜레이를 먹는지 통합 측면의 로깅 존재 검증
- [ ] 16. sseService가 null로 억지 조작되었을 때 NullPointerException을 try-catch가 꿀꺽 삼킬 수 있는 최후 방어벽 검증

### 극단적 타이밍 결함 방어 보장 (4)

- [ ] 17. 큐에서 500개를 빼내는 사이에 중간에 접속자 카운트가 순식간에 0으로 변했다고 하더라도, 일단 뽑힌 것은 무사히(또는 폐기되어도 무방하게) 처리되는가
- [ ] 18. while(true) 무한 루프 내에 메모리 부족(OOM 직전) 현상이 발생할 때 프로그램 전체 다운타임 방지가 되는가 (설정상 catch Exception 기반)
- [ ] 19. 버퍼의 데이터 반환이 매우 느려진(Latency) 경우에도 다음 스케줄 주기 전까지 루프가 블로킹되지 않음을 검증(Mocking sleep)
- [ ] 20. 다회 주기 반복: 연속 5회 호출 시 (접속 0 -> 접속 1 -> 접속 0 -> 접속 3 등) 매 주기별로 최적화가 올바르게 On/Off 플립핑 되는지 동적 행동 검증
